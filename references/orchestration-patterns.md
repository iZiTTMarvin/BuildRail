# Agent 编排模式

> **免责声明**：本文件源自第三方仓库的编排经验整理，其中**「Claude Code 兼容性」章节和「实战案例：Agent Teams」章节**涉及 Claude Code 平台专属概念（subagent、Agent Teams、内置 `Explore`、plugin frontmatter 限制等）。BuildRail 自身不依赖这些机制——BuildRail 的核心命令（`br-bugfix` / `br-iterate` / `br-full-dev` / `run`）都是顺序编排，跨 skill 调用通过返回值完成。文件前半部分的"推荐模式"和"反模式"对所有 markdown 驱动的 agent 通用；后半部分的 Claude Code 章节作为可选参考保留。

本仓库推荐的 agent 编排模式参考目录，以及需要避免的反模式。在添加新的会协调多个专家视角的斜杠命令之前，或在引入"包装"现有专家视角的新专家之前，请先阅读本文件。

核心规则：**用户（或斜杠命令）是编排者。专家视角不调用其他专家视角。** Skill 是专家视角工作流中的必经步骤。

---

## 推荐模式

### 1. 直接调用（无编排）

单个专家视角，单个分析角度，单个产物。这是默认选项，也是成本最低的选项。

```
用户 → 代码审查专家 → 报告 → 用户
```

**适用场景：** 工作内容是对单个产物的单一角度分析，且能用一句话描述清楚。

**示例：**
- "审查这个 PR" → `代码审查专家`
- "找出 `auth.ts` 中的安全问题" → `安全审计专家`
- "结账流程缺少哪些测试？" → `测试工程师`

**成本：** 一次往返。这是所有编排模式的基线成本，任何编排模式都应与它对比。

---

### 2. 单专家视角斜杠命令

一个斜杠命令包装单个专家视角并加载项目的 skill。省去用户每次重复说明工作流的麻烦。

```
/br-review → 代码审查专家（加载 br-review skill） → 报告
```

**适用场景：** 同一个单专家视角调用反复出现，且每次的配置都相同。

**本仓库示例：** `/br-review`、`/br-test`。其他安全/简化/性能类专家视角可按需新增。

**成本：** 与直接调用相同。斜杠命令本质上是一个保存好的提示词。

**反信号：** 如果斜杠命令的内容主要是"决定调用哪个专家视角"，就应该删掉它，让用户直接调用对应专家。

---

### 3. 并行 fan-out + merge

多个专家视角同时处理同一输入，各自产出独立报告。然后由一个合并步骤（在主 agent 的上下文中执行）将结果综合为单一决策。

```
                        ┌─→ 代码审查专家   ─┐
/run → fan out ───┼─→ 安全审计专家  ─┤→ merge → 通过/不通过 + 回滚方案
                        └─→ 测试工程师     ─┘
```

**适用场景：**
- 子任务之间真正独立（无共享可变状态，无顺序依赖）
- 每个子 agent 都能从独立的上下文窗口中获益
- 合并步骤足够小，能装进主上下文
- 实际等待时间是一个考虑因素

**本仓库示例：** `/run`（BuildRail 的执行命令）。

**成本：** N 个并行子 agent 上下文 + 一次合并轮次。高于直接调用，但实际等待时间更短，且每个子 agent 保持专注，产出的报告质量更高。

**采用此模式前的验证清单：**
- [ ] 所有子 agent 能否同时运行而不产生顺序问题？
- [ ] 每个专家视角是否产出不同*类型*的发现，而不是同一发现的不同角度？
- [ ] 合并步骤是否能装进主 agent 的剩余上下文？
- [ ] 用户的等待时间是否足够长，使得并行化有实际感知？

如果任何一个答案是"否"，就退回到直接调用或单专家视角命令。

---

### 4. 用户驱动的顺序管道（斜杠命令序列）

用户按定义好的顺序依次运行斜杠命令，在步骤之间传递上下文（或提交历史）。这里没有编排 agent——用户本身就是编排者。

```
用户依次运行：  /br-plan  →  /br-plan  →  /run  →  /br-test  →  /br-review  →  /run
```

**适用场景：** 工作流有依赖关系（每一步需要上一步的输出），且步骤之间的人类判断能增加价值。

**本仓库示例：** 完整的 规划 → 构建 → 验证 → 审查 → 执行 生命周期。

**成本：** 每步一个子 agent 上下文。编排层零成本，因为没有编排 agent。

**为什么不自动化：** 一个 LLM "生命周期编排器"会（a）在步骤之间丢失细节，因为它必须为交接做摘要；（b）跳过那些能及早发现方向错误的人类检查点；（c）通过复述轮次使 token 成本翻倍。

---

### 5. 研究隔离（上下文保留）

当任务需要阅读大量不应污染主上下文的材料时，派生一个研究子 agent，只返回摘要。

```
主 agent → 研究子 agent（阅读 50 个文件） → 摘要 → 主 agent 继续
```

**适用场景：**
- 主会话需要保持对下游任务的专注
- 调查结果远小于它所消费的输入量
- 决策质量受益于主 agent 在获取结果后有充足的思考空间

**示例：** "在整个代码仓库中找到这个废弃 API 的所有调用点"、"总结这 30 个 ADR 关于缓存的结论"。

**成本：** 一个隔离的子 agent 上下文。只要替代方案是把几百个文件加载进主上下文，这个成本就是值得的。

**在 Claude Code 上，使用内置的 `Explore` subagent**，而不是定义自定义的研究专家。`Explore` 运行在 Haiku 上，被拒绝写入/编辑工具，专门为此模式构建。只有在 `Explore` 不适用时（例如你需要领域特定的系统提示词），才定义自定义研究子 agent。

## Claude Code 兼容性

本目录与具体 harness 无关，但大多数读者会在 Claude Code 上运行它。以下是每种模式如何映射到 Claude Code 的原语——以及平台在哪里替我们强制执行规则。

### 专家视角存放位置

插件 subagent 放在插件根目录的 `agents/` 中。本仓库是一个插件（`.claude-plugin/plugin.json`），所以 `agents/code-reviewer.md`、`agents/security-auditor.md`、`agents/test-engineer.md` 会在插件启用时被自动发现，无需路径配置。

> **BuildRail 说明：** BuildRail 当前暂无独立的 `agents/` 目录。BuildRail 通过 skill 内嵌专家视角（代码审查、安全审计、测试等），未来可拆分为独立的 agent 文件。

### Subagent vs. Agent Teams

Claude Code 有两种并行原语。模式 3（并行 fan-out + merge）映射到 **subagent**。如果你需要能互相交流的队友，使用 **Agent Teams**。

| | Subagent | Agent Teams |
|--|-----------|-------------|
| 协调方式 | 主 agent fan-out，子 agent 只向主 agent 报告 | 队友之间直接发消息，共享任务列表 |
| 上下文 | 每个 subagent 有独立的上下文窗口 | 每个队友有独立的上下文窗口 |
| 适用场景 | 独立任务，产出报告 | 需要讨论的协作工作 |
| 状态 | 稳定 | 实验性——需要 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` |
| 成本 | 较低 | 较高——每个队友是一个独立的 Claude 实例 |

**本仓库的专家视角在两种模式下都能工作。** 当作为 subagent 生成时（例如被 `/run` 调用），它们向主会话报告发现。当作为队友生成时（`使用 security-auditor agent 类型生成一个队友……`），它们可以直接质疑彼此的发现。专家视角的定义是相同的，只是生成上下文不同。

一个细节：专家视角的 frontmatter 中的 `skills` 和 `mcpServers` 字段在作为 subagent 运行时生效，但**在作为队友运行时被忽略**——队友从你的项目和用户设置中加载 skill 和 MCP 服务器，与常规会话相同。如果某个专家视角依赖特定的 skill 或 MCP 服务器，请在会话级别配置它，以确保两种模式都可用。

> **BuildRail 说明：** BuildRail 当前未实现独立 agent 层，Agent Teams 能力依赖 Claude Code 平台支持。

### 平台强制执行的规则

本目录中的两条规则不仅是约定——Claude Code 会强制执行它们：

- **"Subagent 不能生成其他 subagent"**（原文如此）。反模式 B（专家调用专家）和反模式 D（深层专家树）在 Claude Code 上构造上就不可能存在。
- **"不能嵌套 team"**——队友不能生成自己的 team。同样的反模式在 team 层级也被阻止。

这意味着你可以采用本目录中的模式，而不必担心贡献者意外构建出反模式。它们根本无法加载。

### 需要了解的内置 subagent

在定义自定义 subagent 之前，先检查这些是否已覆盖所需角色：

| 内置 | 用途 |
|------|------|
| `Explore` | 只读的代码库搜索和分析。用于模式 5（研究隔离）。 |
| `Plan` | 计划模式中的只读研究。 |
| `general-purpose` | 需要同时进行探索和修改的多步骤任务。 |

不要重新定义这些。你的专家视角（代码审查专家、安全审计专家、测试工程师）是在它们之上叠加的。

### 插件 agent 的 frontmatter 限制

插件 subagent **不支持** `hooks`、`mcpServers` 或 `permissionMode` frontmatter 字段——这些会被静默忽略。如果未来的专家视角需要其中任何一个，用户必须将文件复制到 `.claude/agents/` 或 `~/.claude/agents/` 中。

插件 agent 中**可用**的字段有：`name`、`description`、`tools`、`disallowedTools`、`model`、`maxTurns`、`skills`、`memory`、`background`、`effort`、`isolation`、`color`、`initialPrompt`。如果想优化成本，可以按专家视角设置 `model`（例如 Haiku 用于 `测试工程师` 的覆盖扫描，Sonnet 用于 `代码审查专家`，Opus 用于 `安全审计专家`）。

### 并行生成多个 subagent

在 Claude Code 中，并行 fan-out（模式 3）需要在**单个 assistant 轮次中发出多个 Agent 工具调用**。顺序轮次会导致串行执行。`/run` 命令明确使用了这一点。任何新的编排命令都应如此。

## 实战案例：用 Agent Teams 进行竞争假设调试

这个例子说明何时应该使用 **Agent Teams** 而不是 `/run` 的 subagent fan-out。这两种模式从远处看很相似——都是生成同样的三个专家视角——但价值来源不同。

### 场景

> *结账流程偶尔会在完成前挂起约 30 秒。大约每 50 次会话发生一次。日志中没有错误。上周的发布后开始出现。*

可能的根因（互斥，都符合症状）：

1. 新的支付确认流程中存在竞态条件
2. 某个认证检查偶尔会回退到缓慢的同步网络调用
3. 某个查询缺少索引，随购物车大小而变慢
4. 某个不稳定的第三方 API，其 SDK 在超时前静默重试

单个 agent 会选择第一个看起来合理的理论然后停止调查。`/run` 式的 subagent fan-out 会让每个专家独立报告——但它们的报告互不交集，所以没有什么能排除错误的理论。

这正是 Agent Teams 文档描述的场景：*"多个独立的调查者主动尝试推翻彼此的假设，存活下来的理论更有可能是真正的根因。"*

### 为什么这*不是* `/run` 的工作

| | `/run`（subagent） | Agent Teams |
|--|--------------------|-------------|
| 子 agent 看到 | 相同的 diff，不同角度 | 共享任务列表，彼此的消息 |
| 输出 | 三份独立报告 → 一次合并 | 对抗性辩论 → 共识根因 |
| 适用时机 | 你想要对已知产物的判定 | 你想在多个假设中*找到*那个产物 |

`/run` 是判定；Agent Teams 是调查。

### 设置（一次性，按环境）

Agent Teams 是实验性功能。在 `~/.claude/settings.json` 中：

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

需要 Claude Code v2.1.32 或更高版本。本仓库中的专家视角会被自动拾取——不需要手动编写 team 配置文件。

### 触发提示词

在主导会话中，用自然语言输入：

```
用户报告结账流程在上周发布后间歇性挂起约 30 秒。日志中无错误。

创建一个 agent team 来用竞争假设调试这个问题。生成三个
队友，使用现有的 agent 类型：

  - code-reviewer（代码审查专家） — 调查结账代码路径中的竞态条件和阻塞调用
  - security-auditor（安全审计专家） — 调查认证检查、会话处理以及最近添加的任何同步网络调用
  - test-engineer（测试工程师） — 提出能区分各假设的测试，检查结账流程的覆盖缺口

让他们直接互相发消息来质疑彼此的理论。随着共识的出现更新发现。
只有当两个队友同意能推翻其他假设时才收敛。
```

主导会话生成三个队友，引用现有的专家视角名称。专家视角的正文被**追加**到每个队友的系统提示词中（作为额外指令，叠加在主导会话安装的 team 协调指令之上）；上面的触发提示词成为它们的任务。

### 执行过程

1. 每个队友在自己的上下文窗口中运行，从自己的角度探索代码库。
2. 队友使用 `message` 直接向彼此发送发现。主导会话不需要中转。
3. 共享任务列表显示谁在调查什么——随时可以通过 `Ctrl+T`（进程内模式）或在 tmux 窗格（分屏模式）中查看。
4. 当 `code-reviewer`（代码审查专家）发现一个应该是顺序执行的 `Promise.all` 时，它会发消息给 `security-auditor`（安全审计专家）确认认证调用不是竞态的一部分。`security-auditor` 检查后回复——要么确认竞态是真正的问题，要么提供反面证据。
5. `test-engineer`（测试工程师）为当前最有说服力的理论提出一个聚焦的集成测试，团队用它来验证后再宣布共识。
6. 主导会话综合收敛的发现并呈现给你。

你可以随时通过 `Shift+Down` 切换到任意队友并输入内容——适合纠正走错方向的调查者。

### 何时清理

当调查确定了根因后，告诉主导会话：

```
清理团队
```

始终通过主导会话清理，而不是通过队友（文档说明：队友缺少完整的团队上下文来执行清理）。

### 成本预期

三个 Sonnet 队友运行约 10–15 分钟的调查，成本明显高于同样三个专家视角作为 subagent 被 `/run` 生成的成本。理由是*结论质量*——对于生产环境调试，错误的修复代价很高，多花的 token 是划算的。对于常规 PR 审查，还是用 `/run`。

### 此场景中的反模式

**不要**把它重建为一个 fan-out subagent 的 `/debug` 斜杠命令。Subagent 不能互相发消息——你会失去使这个模式生效的对抗性辩论。如果某个工作流反复出现，把上面的触发提示词文档化为一个代码片段，而不是包装成误用 subagent 的斜杠命令。

### 何时*不应*使用 Agent Teams

- 对已知 diff 的生产就绪判定 → 使用 `/run`（subagent）。
- 单个专家视角对单个产物 → 直接调用。
- 顺序生命周期（规划 → 构建） → 用户驱动的斜杠命令（模式 4）。
- 读密集型研究，摘要很小 → 内置 `Explore` subagent。

只有当队友**需要**互相质疑才能产出正确答案时，才使用 Agent Teams。

## 反模式

### A. 路由专家（"元编排器"）

一个专家视角，其工作是决定调用哪个其他专家视角。

```
/work → 路由专家 → "这需要审查" → 代码审查专家 → 路由专家（复述） → 用户
```

**为什么失败：**
- 纯路由层，没有领域价值
- 增加两个复述跳 → 信息丢失 + 大约 2 倍 token 成本
- 用户本来就知道他们想要审查；可以直接调用 `/br-review`
- 复制了斜杠命令和 `CLAUDE.md` 中的意图映射已经完成的工作

**替代方案：** 添加或优化斜杠命令。在 `CLAUDE.md` 中文档化意图 → 命令映射。

---

### B. 专家调用其他专家

一个 `代码审查专家` 在看到认证代码时内部调用 `安全审计专家`。

**为什么失败：**
- 专家视角被设计为产出单一角度的分析；链式调用违背了这个设计
- 调用方传递的摘要会丢失被调用方需要的上下文
- 失败模式倍增（谁的输出格式优先？谁的规则适用？）
- 对用户隐藏了成本

**替代方案：** 让调用方在报告中*推荐*后续审计。用户或斜杠命令运行第二轮。

---

### C. 复述型顺序编排器

一个 agent 代替用户依次调用 `/br-plan`、`/br-plan`、`/run` 等。

**为什么失败：**
- 丢失了那些能及早发现方向错误的人类检查点
- 每次交接都要做上下文摘要——长管道中累积漂移
- token 成本翻倍：每步都是编排器轮次 + 子 agent 轮次
- 在最需要判断的节点上剥夺了用户的主动权

**替代方案：** 让用户作为编排者。在 `README.md` 中文档化推荐顺序，让用户自己调用。

---

### D. 深层专家树

`/run` 调用一个 `pre-run-coordinator`，后者调用一个 `quality-coordinator`，后者再调用 `代码审查专家`。

**为什么失败：**
- 每一层增加延迟和 token，没有决策价值
- 调试变成多层级调查
- 叶子专家视角经过多次摘要后丢失上下文

**替代方案：** 编排深度最多为 1（斜杠命令 → 专家视角）。合并在主 agent 中完成。

## 决策流程

当考虑一个新的编排工作流时，按此流程走：

```
工作是否是对单个产物的单一角度分析？
├── 是 → 直接调用。停止。
└── 否 → 同样的组合是否会重复出现？
         ├── 否 → 临时直接调用。停止。
         └── 是 → 子任务是否独立？
                  ├── 否 → 用户驱动的顺序斜杠命令（模式 4）。
                  └── 是 → 并行 fan-out + merge（模式 3）。
                           用上面的验证清单校验。
                           如果任何检查项不通过 → 退回到单专家视角命令（模式 2）。
```

## 何时向本目录添加新模式

只有在满足以下条件后才添加新条目：

1. 你在实际工作中至少使用过该模式两次
2. 你能指出本仓库中演示它的具体产物
3. 你能解释为什么现有模式不适用
4. 你能描述它的反模式阴影（人们会误建什么来替代它）

过早的目录条目会变成没人遵循的愿景式文档。
