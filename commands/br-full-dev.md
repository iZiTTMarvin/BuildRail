# /br-full-dev — 完整开发流程

从需求构思到功能发布的端到端标准工作流，串联整个 BuildRail 体系。
**重要：本工作流是【路径 A：全自动】。零参数、零门禁，一路跑到底。中间任何阶段都不停下等待用户确认——设计文档生成后直接进计划，计划生成后直接进执行。这是 br-full-dev 的定义本身。**（想分步掌控？用 `/idea` → `/br-plan` → `/run` → ... 见 `shared/two-paths.md`）

**唯一会停的情况**：连续 3 次验收失败（已超出"自动"能解决的范围），届时停下，用 `/br-status` 查看诊断。

**用户需求：** $ARGUMENTS

---

## 阶段 1：需求构思与设计

根据用户的需求复杂度，内部自主决定路由到适当的探索 skill，产出设计/需求文档。

1. **自动路由判断**：
   - 目标是全新项目或系统级大重构 → 调用 `/br-office-hours`（产出 `-design.md`）。
   - 目标是在已有系统上新增/增强具体功能 → 调用 `/br-brainstorming`（产出 `-requirement.md`）。
   > 🟡 通知用户路由选择结果，并立即开始提问。
2. **需求挖掘**：执行对应 skill 的交互式追问，直到信息足够。
3. **文档生成**：将提炼好的文档存放到 `.buildrail/idea/` 目录。
4. **状态置为 APPROVED**（全自动，不等确认）。通知一句"🟡 设计文档已生成（{路径}），继续推进计划阶段"。然后**立即进入阶段 2**。

---

## 阶段 2：计划生成与拆分

在设计文档获得 APPROVED 后，调用 `/br-plan` 中的核心逻辑，制定可执行任务列表。

1. **范围挑战**：调用 `/br-scope-check`，对设计文档进行 6 项审查。
   - 发现 HIGH 级别警告时，尝试修改设计解决。最多进行 2 轮，如果依然存在 HIGH，则作为风险 Tradeoff 记录并继续，不要死循环。
2. **任务拆分**：调用 `/br-task-breakdown`，将设计拆分为具体的垂直切片任务，强制要求包含验收标准。
3. **文档生成**：将计划输出到 `.buildrail/plans/` 目录。
4. **状态置为 APPROVED**（全自动，不等确认）。通知一句"🟡 拆分计划已生成（{路径}），继续推进执行阶段"。然后**立即进入阶段 3**。

---

## 阶段 3：执行与验证循环

在计划文档获得 APPROVED 后，嵌入并调用 `/run` 的核心执行逻辑。

1. **任务循环**：针对计划中的每一个任务（Task）：
   - **实现**：编写实际代码。
   - **验收**：调用 `/br-verify` 逐条核对验收标准。
   - **调试失败**：如果验收 FAIL，自动调用 `/br-debug` 排查修复。同一任务最多重试 3 次，第 3 次失败时将其标记为 `SKIPPED` 并记录原因，强制进入下一个任务。
   - **原子提交**：每个验证通过的任务立即执行 `git commit`。
2. **全量验证**：所有任务遍历完成后，自动检测并运行项目的全局测试和构建命令（如 `npm test`, `pytest`, `npm run build`）。如果全量测试失败，必须尝试修复。

---

## 阶段 4：综合代码审查

在代码全部实现并在全量测试通过后，执行代码库级别的质量门禁。

1. **五轴审查**：调用 `/br-review` 对本次的所有 Diff 进行（正确性、可读性、架构、安全、性能）五轴审查。
2. **问题处理**：
   - 发现 **Critical / HIGH** 问题：自动进行修复，然后重跑局部测试，通过后才能放行。
   - 发现 **Medium / Low / Nit** 问题：如果容易修就顺手修掉，如果复杂则记录为技术债，继续流程。

---

## 阶段 5：发布

通过全部门禁后，收尾并发布功能。

1. 调用 `/br-ship` skill。
2. 将完成的功能点汇总更新到 `CHANGELOG.md`。
3. 将最终代码 `git push` 到远程仓库。

## 结束通知
向用户输出完整的项目总结："🎉 端到端开发流程已完成！[特性名称] 已经设计、拆分、实现、验证并发布。"
