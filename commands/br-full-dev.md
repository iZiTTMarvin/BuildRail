# /br-full-dev — 完整开发流程

从需求构思到功能发布的端到端标准工作流，串联整个 BuildRail 体系。
**重要：本工作流是【路径 A：全自动】。**开发阶段（设计→计划→编码→审查）零参数、零门禁、一路跑到底——设计文档生成后直接进计划，计划生成后直接进执行，执行完直接进审查，中间不停下等待用户确认。这是 br-full-dev 的核心卖点。**

**但在审查通过、即将发布（commit/push）时，br-full-dev 会停下来汇报成果，让用户决定是否提交** —— 因为提交和推送是对外、难撤销的操作，属于合格工作流的 human-in-the-loop 底线，不是对"全自动"的削弱。详见下方阶段 5。（想全程分步掌控？用 `/idea` → `/br-plan` → `/run` → ... 见 `shared/two-paths.md`）

**会停下等待的情况**：
- 连续 3 次验收失败（已超出"自动"能解决的范围），届时停下，用 `/br-status` 查看诊断。
- 阶段 5 发布前：审查通过后汇报成果，询问用户是否提交推送。

## 阶段 1：需求构思与设计

根据用户的需求复杂度，内部自主决定路由到适当的探索 skill，产出设计/需求文档。

1. **自动路由判断**：
   - 目标是全新项目或系统级大重构 → 调用 `/br-office-hours`（产出 `-design.md`）。
   - 目标是在已有系统上新增/增强具体功能 → 调用 `/br-brainstorming`（产出 `-requirement.md`）。
   > 🟡 通知用户路由选择结果，并立即开始提问。
2. **需求挖掘**：执行对应 skill 的交互式追问，直到信息足够。
3. **文档生成**：将提炼好的文档存放到 `.buildrail/idea/` 目录。
4. **状态置为 APPROVED**（全自动，不等确认）。通知一句"🟡 设计文档已生成（{路径}），继续推进计划阶段"。然后**立即进入阶段 2**。

---

## 阶段 2：计划生成与拆分

在设计文档获得 APPROVED 后，调用 `/br-plan` 中的核心逻辑，制定可执行任务列表。

1. **范围挑战**：调用 `/br-scope-check`，对设计文档进行 6 项审查。
   - 发现 HIGH 级别警告时，尝试修改设计解决。最多进行 2 轮，如果依然存在 HIGH，则作为风险 Tradeoff 记录并继续，不要死循环。
2. **任务拆分**：调用 `/br-task-breakdown`，将设计拆分为具体的垂直切片任务，强制要求包含验收标准。
3. **文档生成**：将计划输出到 `.buildrail/plans/` 目录。
4. **状态置为 APPROVED**（全自动，不等确认）。通知一句"🟡 拆分计划已生成（{路径}），继续推进执行阶段"。然后**立即进入阶段 3**。

---

## 阶段 3：执行与验证循环

在计划文档获得 APPROVED 后，嵌入并调用 `/run` 的核心执行逻辑。

1. **任务循环**：针对计划中的每一个任务（Task）：
   - **实现**：编写实际代码。
   - **验收**：调用 `/br-verify` 逐条核对验收标准。
   - **调试失败**：如果验收 FAIL，自动调用 `/br-debug` 排查修复。同一任务最多重试 3 次，第 3 次失败时将其标记为 `SKIPPED` 并记录原因，强制进入下一个任务。
   - **标记完成（不提交）**：每个验收通过的任务只更新 state，**不执行 git commit**。代码改动保留在工作区，等阶段 4 审查通过、阶段 5 用户确认后，由 `/br-ship` 统一提交。把 commit 嵌在执行循环里会让阶段 4 的审查沦为事后擦屁股。
2. **全量验证**：所有任务遍历完成后，自动检测并运行项目的全局测试和构建命令（如 `npm test`, `pytest`, `npm run build`）。如果全量测试失败，必须尝试修复。

---

## 阶段 4：综合代码审查

在代码全部实现并在全量测试通过后，执行代码库级别的质量门禁。

1. **五轴审查**：调用 `/br-review` 对本次的所有 Diff 进行（正确性、可读性、架构、安全、性能）五轴审查。
2. **问题处理**：
   - 发现 **Critical / HIGH** 问题：自动进行修复，然后重跑局部测试，通过后才能放行。
   - 发现 **Medium / Low / Nit** 问题：如果容易修就顺手修掉，如果复杂则记录为技术债，继续流程。

---

## 阶段 5：汇报与发布（human-in-the-loop）

> **产品底线**：前 4 个阶段（设计→计划→编码→审查）全自动，这是 br-full-dev 的核心卖点。
> 但**提交（commit）和推送（push）是对外、难撤销的操作，必须由用户确认后才执行**——
> 这是合格工作流的底线，不是对"全自动"的削弱。详见 `skills/br-ship/SKILL.md` 的第零步。
>
> **为什么不自动 commit？** 用户跑完后可能还想微调（改文案、调样式）。自动 commit 会挡住这些微调，
> 让用户被迫重新组织提交。所以把"何时提交"的决定权留给用户。

### 1. 汇报本次成果（排版清晰）

把本次完成的工作结构化呈现给用户：

- **做了什么**：实现/修复的功能点（X / Y / Z），简明列出。
- **改了哪些文件**：文件列表 + 每个文件的一句话改动概述。
- **审查结果**：五轴审查各轴结论，有无遗留问题或技术债。
- **当前状态**：明确告知"以上改动都还在缓存区（工作区），**尚未提交**"。

### 2. 询问用户是否提交

汇报完后询问：**"以上改动都还在缓存区。是否需要我帮你提交并推送？"**

- **用户不确定（尤其新手）** → 先用大白话解释 commit / push 是什么、做了会怎样，再问一次。
- **用户说"要"** → **调用 `/br-ship`** 完成提交与推送（br-ship 内部会再次确认 commit message、执行 CHANGELOG 更新 + commit + push）。
- **用户说"不要 / 我再看看"** → 停止，不调用 br-ship。改动留在工作区，提示用户：*"改好可用 `/br-ship` 提交；想微调用 `/br-iterate`。"*
- **用户想先改某处** → 建议 `/br-iterate <改动描述>` 微调，改完再 `/br-ship`。

> **复用 /br-ship，不重写提交逻辑**：阶段 5 只负责"汇报 + 询问"，真正的提交动作交给 `/br-ship`。
> 这样避免逻辑重复，且 br-ship 单独被调用时行为一致。

## 结束通知

根据用户的选择输出总结：

- **用户同意提交**：`/br-ship` 执行完后，"🎉 端到端开发流程已完成！[特性名称] 已设计、拆分、实现、验证并发布。"
- **用户暂不提交**："✅ 开发部分已完成（设计→编码→审查全通过），改动保留在工作区未提交。需要提交时运行 `/br-ship`，想微调运行 `/br-iterate`。"
