# /br-debug — 单任务深度调试

对一个任务（通常是被 `/run` 跳过的、或验收失败的）做系统化深度调试：复现 → 定位根因 → 修 → 加防护测试。比 `/run` 的自动重试更彻底。

**任务 ID 或问题描述：** $ARGUMENTS

---

## 什么时候用这个命令

- `/run` 报告某任务连续 3 次验收失败被跳过，你想单独深入排查这一个
- 你看到 `/br-status` 里的诊断后，想针对某个失败任务精准修复
- 修复后用 `/run 从 task-NNN 开始` 补做后续任务

用法示例：
```
/br-debug task-002
/br-debug task-002 限流计数器在并发下丢更新
```

---

## 执行

读取 `skills/br-debug/SKILL.md`，按其 7 步系统化调试流程执行：
1. 复现失败（构造最小失败用例）
2. 分类失败（语法/类型/逻辑/依赖/环境）
3. 定位（缩小问题范围，必要时 `git bisect`）
4. 精简（去掉无关代码，只保留触发 bug 的最小代码）
5. 修根因（不是修症状——问"为什么会这样"直到找到真正原因）
6. 加防护（写一个专门捕获这个 bug 的测试）
7. 端到端验证

---

## 初始化运行状态

按 `shared/state-schema.md` 覆盖式写入 `.buildrail/state.json`：
- `run`：{id: 当前时间 YYYY-MM-DD-HHMM, command: "br-debug", path: "step", started_at, status: "running"}
- `phase`：{current: "debug", label: "深度调试"}

如果用户指定了 task-id（如 `task-002`），从 `.buildrail/state.json` 或最新计划文件读取该任务的描述和验收标准，作为调试上下文。

---

## 重试限制

按 skill 的"重试限制"：**同一个问题最多修 2 次**。
- 第 1 次修失败 → 分析为什么没修好，换方向
- 第 2 次还失败 → **停止**，输出诊断报告，让用户决定

---

## 收尾

**修复成功**：输出修复总结，更新 state.json 对应任务状态为 `done`、`attempts` 为实际次数。

输出：
> "✅ 调试完成。task-002 已修复并加了防护测试。
> **下一步**：可运行 `/run 从 task-002 开始` 继续执行后续任务，或 `/br-verify task-002` 重新验收。"

**修复失败（2 次未解决）**：输出诊断报告（见 skill 的"诊断报告格式"），更新 state.json 对应任务 `status: failed`、`failure` 字段。

输出：
> "❌ task-002 调试失败（2 次未解决）。
> **诊断报告**：{根因分析 + 建议}
> 你可以：手动介入修复，或跳过此任务继续后续。"

---

## 异常处理

| 场景 | 处理 |
|------|------|
| 用户没给 task-id 且无 state.json | 提示"请指定要调试的任务，如 `/br-debug task-002`" |
| 指定的 task-id 在计划中不存在 | 提示"没有找到 task-002，请检查 `/br-status` 里的任务列表" |
| bug 无法稳定复现 | 按 skill 的"偶发 bug 复现策略"处理（CI 环境、状态依赖、真随机） |
| 调试中发现是设计问题 | 建议用户回到 `/br-scope-check` 或 `/br-office-hours` |

## 语气风格

- 像一个资深工程师在排查问题——有理有据，不猜
- 复现/定位/修复每一步都说明在做什么、为什么
- 用中文，用"你"不用"您"
