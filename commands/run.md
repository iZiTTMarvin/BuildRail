# /run — 按计划执行

从实现计划中逐个执行任务，每个任务走"实现→验收→提交"循环。

**用户需求：** $ARGUMENTS

---

## 执行循环

```
读取计划
    │
    ├─ 无 APPROVED 计划 → 提示先跑 /br-plan
    │
    └─ 找到 → 逐个执行任务
        │
        ┌──────────────────────────────────────┐
        │                                      │
        │  实现 ──→ 验收 ──→ 通过 ──→ 提交     │
        │    │                 │               │
        │    │                 └──────────┐    │
        │    │                            │    │
        │    └── 失败 → /br-debug 修复    │    │
        │              │                  │    │
        │              ├─ 修复成功 → 验收  │    │
        │              └─ 2次失败 → 跳过   │    │
        │                       │          │    │
        │                       └──────────┘    │
        │                                      │
        │              下一个任务 ◄─────────────┘
        │                                      │
        └──────────────────────────────────────┘
                    │
                    └─ 全部完成 → 总结
```

---

## 第一步：读取计划

扫描 `.buildrail/plans/` 目录，找到最新的 APPROVED 计划：

```bash
ls -lt .buildrail/plans/*-plan.md 2>/dev/null
```

**判断**：

| 情况 | 处理 |
|------|------|
| 找到 APPROVED 计划 | 读取，继续 |
| 找到 DRAFT 计划 | 🟡 提示："计划还是 DRAFT 状态。请先确认（说'确认'），或直接继续（说'继续'）。" |
| 无计划 | 提示："没有找到实现计划。请先运行 `/br-plan` 生成计划。" |

读取计划内容，解析任务列表。提取每个任务的：
- 编号（task-001 等）
- 描述
- 验收标准
- 涉及文件
- 依赖关系
- 预估大小

**如果用户指定了起始任务**（如"从 task-003 开始"），跳到该任务。

---

## 第二步：检测项目环境

```bash
# 检测验证命令
cat package.json 2>/dev/null | grep -E '"(test|lint|build|typecheck)"' || true
cat pyproject.toml 2>/dev/null | grep -E '(test|lint|build)' || true
cat Makefile 2>/dev/null | head -20 || true
```

记录：
- 测试命令（`npm test` / `pytest` / `make test` 等）
- 构建命令（`npm run build` / `python -m build` 等）
- Lint 命令（如果有）

---

## 第三步：逐个执行任务

对每个任务，按以下流程执行：

### 3.1 检查依赖

如果任务有依赖，确认依赖任务已完成（在任务列表中标记为 ✅）。
如果依赖未完成 → 跳过此任务，先执行依赖任务。

### 3.2 实现

按任务描述编写代码。

**实现原则**（来自 agent-skills incremental-implementation）：

1. **简单优先**：写代码前问"最简单能用的方案是什么？"
2. **范围纪律**：只改任务要求的文件，不要"顺便"改其他东西
3. **一次一件事**：不要在一个任务里混多个不相关的改动
4. **保持可编译**：每一步改完，项目必须能构建

**进度通知**：
> 🟡 开始执行 task-001: {任务描述}

### 3.3 验收

调用 `br-verify` skill 验证验收标准。

**验收结果处理**：

| 结果 | 处理 |
|------|------|
| 全部 PASS | ✅ 进入提交 |
| 有 FAIL | 进入调试重试 |

### 3.4 调试重试（验收失败时）

**第 1 次失败**：
> 🟡 task-001 验收失败（1/3 未通过），正在调试...

调用 `br-debug` skill 进行系统化调试：
1. 复现失败
2. 定位根因
3. 修复代码
4. 重新验收

**第 2 次失败**：
> 🟡 task-001 验收再次失败（2/3 未通过），再试一次...

换方向修复（不在同一个方向继续细调）。

**第 3 次失败**：
> ❌ task-001 连续 3 次验收失败。已跳过。

标记任务为 ⏭️ SKIPPED，附失败原因，继续下一个任务。

### 3.5 提交

验收通过后，提交代码：

```bash
git add {涉及的文件}
git commit -m "feat: {任务描述}"
```

在任务列表中标记为 ✅ DONE。

**进度通知**：
> ✅ task-001 完成。进度: 1/{总任务数}

### 3.6 下一个任务

回到 3.1，执行下一个任务。

---

## 第四步：全量验证

所有任务执行完后，运行全量验证：

```bash
# 运行全量测试
npm test 2>&1 || pytest 2>&1 || make test 2>&1 || echo "无测试命令"

# 运行构建
npm run build 2>&1 || python -m build 2>&1 || echo "无构建命令"

# 运行 lint（如果有）
npm run lint 2>&1 || ruff check . 2>&1 || echo "无 lint 命令"
```

**结果处理**：

| 结果 | 处理 |
|------|------|
| 全部通过 | ✅ 进入总结 |
| 测试失败 | 🟡 尝试修复回归问题 |
| 构建失败 | 🟡 检查类型错误或编译问题 |
| 修复后仍失败 | 记录问题，不阻塞总结 |

---

```markdown
**执行时间：** {开始时间} - {结束时间}

### 任务完成情况

| 任务 | 状态 | 说明 |
|------|------|------|
| task-001: {描述} | ✅ DONE | |

- 测试：✅ 通过 / ❌ 失败
- 构建：✅ 通过 / ❌ 失败
- Lint：✅ 通过 / ❌ 失败

{如果有跳过的任务或验证失败，列出}
```

## 异常处理

| 场景 | 处理方式 |
|------|----------|
| 用户中途说"停止" | 保存当前进度，输出已完成任务的总结 |
| 用户说"跳过当前任务" | 标记为 ⏭️ SKIPPED，继续下一个 |
| 实现过程中发现设计有问题 | 记录问题，建议用户回到 `/br-scope-check` 或 `/br-office-hours` |
| git commit 失败 | 检查原因（冲突、hook），修复后重试一次 |
| 任务之间有文件冲突 | 按依赖顺序执行，不做并行 |
| 所有任务都跳过了 | 输出失败报告，建议用户检查计划质量 |

## 语气风格

- 进度通知用 🟡/✅/❌，简洁一句
- 不要解释内部流程（用户不需要知道 br-debug 跑了几步）
- 用中文，用开发者熟悉的语言
- 不要用"您"，用"你"
