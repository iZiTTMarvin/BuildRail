# /br-iterate — 快速迭代

用于已有功能的小改动、性能优化、文案或配置调整。
特点：跳过复杂的前期设计和深度审查，保留核心的 TDD 测试和自动发布。
**注意：本工作流有强硬的范围门控（Light Impact Gate），超出范围会自动拦截并升级。**

**改动描述：** $ARGUMENTS

---

## 初始化运行状态

按 `shared/state-schema.md` 覆盖式写入 `.buildrail/state.json`：
- `run`：{id: 当前时间 YYYY-MM-DD-HHMM, command: "br-iterate", path: "step", started_at, status: "running"}
- `phase`：{current: "iterate", label: "快速迭代", entered_at}
- `tasks`：根据改动拆分阶段任务（范围门控 / TDD 迭代 / 轻量审查 / 发布），初始全 `pending`

每个阶段推进时更新 `phase.label` 和对应任务状态。

---

## 阶段 0：范围预检 (Light Impact Gate)

在写任何代码之前，必须快速分析用户的改动目标。如果预估改动满足以下 **任何一项** 高风险特征，或超出了量化阈值，**必须自动拦截并建议升级到 `/br-full-dev`**。

**量化阈值限制（必须全部符合）：**
- 代码改动预估 < 100 行
- 涉及文件数 <= 5 个
- 涉及模块数 <= 2 个
- 绝对不引入新的第三方依赖
- 不修改现有的公开 API 契约或数据库 Schema

**高风险拦截（命中任一立即升级）：**
- 涉及核心业务（认证、鉴权、支付、交易计算）
- 新增独立的页面、路由、微服务或复杂视图组件
- 修改多处依赖的共享 Utils 或核心配置文件

> 🟡 **通过检查**：向用户简要确认"改动在安全范围内，开始迭代"，并自动继续。
> 🔴 **超限拦截**：暂停执行并回复："该改动超出了快速迭代的范围（具体原因：...），为了保证项目架构的安全，强烈建议改用 `/br-full-dev` 走完整设计和审查流程。你希望转为 full-dev，还是强制继续？"（必须等待用户答复）

---

## 阶段 1：分析与准备

1. **理解上下文**：快速分析改动影响了哪些文件。
2. **寻找测试**：找到相关的现有测试文件；如果该功能缺失测试，计划在阶段 2 补充。

---

## 阶段 2：TDD 迭代

1. **写/更新测试**：调用 `/br-test` 编写或修改能覆盖本次变更的测试用例（先确保测试反应了新需求）。
2. **实现改动**：编写实现代码，确保新加的测试通过。
3. **全量验证**：自动扫描项目内的全局测试命令（`npm test` / `pytest` 等）并执行。要求必须 100% PASS。

---

## 阶段 3：轻量审查与 QA

小改动默认跳过重度架构审查，但存在以下例外：
- 如果改动后（Diff）发现实际修改范围比预期大，调用 `/br-review` 做一次代码审查。
- 如果改动涉及用户界面 (UI) 的样式或文案调整，调用 `/br-verify` (扮演 QA) 根据上下文核对一遍。
  - **涉及前端时**，按 `references/accessibility-checklist.md` 核对键盘导航、屏幕阅读器、对比度、ARIA 等基本项。重点关注：纯图标按钮的 `aria-label`、表单字段的 `<label>` 关联、焦点轮廓是否被移除。

---

## 阶段 4：汇报与发布（human-in-the-loop）
调用 `/br-ship` skill。br-ship 会：
1. **汇报本次改动**（改了什么、改了哪些文件、验证结果）
2. **询问你是否提交**："以上改动还在缓存区，是否需要我帮你提交并推送？"
   - 新手不懂"提交"是什么 → AI 用大白话解释 commit/push 含义，再问一次
   - 你说"要" → br-ship 确认 commit message 后 commit + push
   - 你说"不要 / 我再看看" → 改动留在工作区，提示"改好用 `/br-ship` 提交"

> **与 br-full-dev 阶段 5 一致**：小改动也要停下来问，保持产品底线统一——commit/push 是对外操作，必须用户点头。

## 结束通知

根据用户在阶段 4 的选择：
- **用户同意提交**："✅ 快速迭代已完成：[一句话总结改动点]，已提交并推送。"
- **用户暂不提交**："✅ 改动已完成并验证通过，保留在工作区未提交。需要提交时运行 `/br-ship`。"

后续维护：发现 bug 用 `/br-bugfix`，再做新功能用 `/br-full-dev`（全自动）或 `/idea`（分步）。

**运行结束写 state.json**：`run.status: completed`、`run.updated_at`、`global_check` 全量验证结果（阶段 2 第 3 步的 `npm test` / `pytest` 等结果）。
