# Design: BuildRail `/idea` + 底层 Skill 架构

Generated by /office-hours on 2026-05-29
Branch: unknown
Repo: xdev-main/BuildRail
Status: DRAFT
Mode: Builder

## Problem Statement

中文开发者需要一个 AI 工作流框架，其中第一步是"搞清楚用户到底想做什么"。现有方案（gstack office-hours、superpowers brainstorm、agent-skills interview-me / idea-refine）各有亮点但不完全适配中文开发者场景。BuildRail 需要一个智能分流 skill `/idea` 来辅助选择，以及两个独立的底层 skill 来处理大方向和小功能两种场景。

## What Makes This Cool

三个 skill 都是独立可用的。`/idea` 是智能分流器，`/br-office-hours` 和 `/br-brainstorming` 是具体执行者。用户可以跳过分流器直接用底层 skill。每个 skill 都是从成熟项目中提取核心逻辑后重新设计的，去掉 telemetry/YC 推销等个人用户不需要的包袱，保留追问质量。

## Constraints

- 面向中文开发者，所有交互和输出用中文
- 个人使用为主，不需要 telemetry、analytics、跨模型审查等企业级功能
- 复用已验证的核心逻辑（gstack office-hours 的追问模式、superpowers brainstorm 的流程、agent-skills interview-me 的意图挖掘）
- 输出放在项目目录 `.buildrail/idea/` 下
- 主流 agent（Claude Code、OpenCode、Windsurf 等）都支持 `/skill-name` 直接调用 skill，不需要额外的 command 层

## Premises

1. 用户有时清楚自己要做什么（直接用 `/br-office-hours` 或 `/br-brainstorming`），有时不清楚（用 `/idea` 辅助分流）
2. 大方向项目需要深度追问（类似 office-hours），小功能需要快速确认意图 + 技术讨论（类似 brainstorm + interview-me）
3. 三个 skill 相互独立，不形成依赖链
4. 核心提问逻辑从成熟项目提取，但交互风格、输出格式按 BuildRail 风格重新设计

## 核心架构

```
BuildRail/
├── skills/
│   ├── idea/                    ← 智能分流 skill（用户入口）
│   │   └── SKILL.md
│   ├── br-office-hours/         ← 大方向 skill（基于 gstack office-hours）
│   │   └── SKILL.md
│   └── br-brainstorming/        ← 小功能 skill（基于 brainstorm + interview-me）
│       └── SKILL.md
└── shared/
    └── output-format.md         ← 共享的输出格式定义
```

**安装方式（后续实现）**：
```bash
npm install -g buildrail
# 在项目中初始化
buildrail init  # 或 br init
# 交互式选择：Claude Code / OpenCode / Windsurf 等
# 自动安装 skill 到对应 agent 的 skill 目录
```

参考项目：Trellis（https://github.com/mindfold-ai/Trellis）的安装模式。

## Approach: 混合方案（已确认）

### 1. `/idea` Skill — 智能分流器

**职责**：分析用户输入，判断该用大方向还是小功能模式，自动路由到对应的 skill。

**分流逻辑**：基于 LLM 语义理解判断，不是关键词匹配。信号词仅供参考理解两类模式的边界。

**判断原则**：
- 如果用户描述涉及"从无到有创建"或"改变系统整体结构"→ 大方向（`/br-office-hours`）
- 如果用户描述涉及"在现有系统上增/删/改具体功能"→ 小功能（`/br-brainstorming`）
- 信号冲突时优先大方向（宁可多问几个问题，也不要把大方向误判为小功能）
- 无法判断时：用 AskUserQuestion 问用户"这是一个全新项目还是功能增强？"

**异常处理**：
- 用户输入极其模糊（如"帮我改进一下代码"）→ 追问一个开放式问题（"能具体说说想改进哪方面吗？"），最多追问 2 次后进入小功能模式（降级策略）
- 用户输入为英文 → 正常分流，用英文追问，输出文档仍用中文
- 项目目录为空或无 git 记录 → 偏向大方向模式

**不产生文档**：`/idea` 只做分流，不产生任何输出文件。真正的需求/设计文档由底层 skill 产生。

### 2. `br-office-hours` Skill — 大方向探索

**来源**：gstack office-hours（Startup Mode + Builder Mode 的核心提问逻辑）

**提取的逻辑**：
- 6个强制追问的核心思想（需求真实性、现状替代品、具体用户、最小切入点、用户观察、未来适配性）
- 前提挑战（Phase 3）
- 方案对比（Phase 4）— 强制 2-3 个方案 + 推荐

**去掉的功能**：
- YC 推销、telemetry、gbrain sync
- 跨模型审查（Codex second opinion）
- Builder Profile 追踪
- Landscape Awareness（外部搜索）
- 视觉设计探索（HTML wireframe）
- 复杂的 session 管理

**简化流程**（7步）：

1. **理解项目上下文**（仅限本地文件，不做外部搜索）
   - 读 README.md、CLAUDE.md（如果存在）
   - 扫目录结构和 `git log --oneline -10`
   - 如果项目目录为空或无 README → 跳过，直接进追问

2. **深度追问**（5-8个问题，一次一个，用 AskUserQuestion）
   - 每个问题附带一个猜测（参考 interview-me 的 guess 模式）
   - 核心维度和示例问题：
     - **需求真实性**："你做这个东西，是为了解决自己遇到的问题，还是觉得别人可能需要？能举个具体场景吗？"
     - **现状替代品**："现在没有这个工具的时候，你是怎么做的？手动？用其他工具？"
     - **具体用户**："你想象中第一个用这个的人是谁？能描述一下他的工作场景吗？"
     - **最小切入点**："如果只能做这个项目的一个功能，哪个最关键？"
     - **核心约束**："有什么是你绝对不想做的？或者必须满足的硬性条件？"
   - 根据用户回答的充分程度灵活选择追问维度，不强制全部覆盖

3. **前提确认**（列出 2-4 个前提）
   - 格式："PREMISES: 1. [陈述] — 同意/不同意？"
   - 用户不同意 → 修正理解，回到追问（最多回退 1 次）

4. **方案对比**（2-3个方案 + 推荐）
   - 每个方案：名称、概述、优缺点、工作量估计
   - 必须包含一个"最小可行方案"

5. **用户选择方案**

6. **写设计文档**（保存到 `.buildrail/idea/YYYY-MM-DD-<topic>-design.md`）
   - 自动创建 `.buildrail/idea/` 目录（如果不存在）

7. **用户确认文档**
   - 用户确认后状态改为 APPROVED
   - 提示："设计文档已保存到 [路径]。是否开始后续流程？"

**异常处理**：
- 用户中途放弃 → 保存当前进度为 DRAFT 状态的文档，标注"未完成"
- 用户回答自相矛盾 → 直接指出矛盾点，让用户澄清
- `.buildrail/idea/` 无写入权限 → 降级到项目根目录，提示用户

**输出格式**（.buildrail/idea/YYYY-MM-DD-<topic>-design.md）：
```markdown
# 设计文档：<标题>

生成时间：YYYY-MM-DD
模式：大方向探索
状态：DRAFT / APPROVED

## 问题陈述
## 目标用户与核心场景
## 约束条件
## 前提
## 方案对比
### 方案 A：<名称>
### 方案 B：<名称>
## 推荐方案及理由
## 实施步骤概要
## 开放问题
```

### 3. `br-brainstorming` Skill — 小功能探索

**来源**：superpowers brainstorm + agent-skills interview-me

**提取的逻辑**：
- brainstorm 的一次问一个问题 + 2-3 个方案对比 + 设计呈现流程
- interview-me 的意图挖掘（"用户真正想要的 vs 以为应该要的"）
- brainstorm 的 YAGNI 原则和增量验证

**去掉的功能**：
- Visual Companion（浏览器可视化）
- writing-plans skill 调用
- spec-document-reviewer-prompt（subagent 审查）

**简化流程**（6步）：

1. **探索项目上下文**（快速，仅限本地文件）
   - 扫文件结构、最近的改动（`git diff --stat HEAD~5`）
   - 如果是已有项目 → 理解现有架构模式
   - 如果是空项目 → 跳过

2. **确认真实意图**（2-3个问题，"guess + question"模式）
   - 示例交互：
     > Agent: 你说想"加个搜索功能"，我猜你是想让用户在列表页快速找到特定数据——对吗？
     > User: 不完全是，我是想让用户能保存常用的搜索条件
     > Agent: 明白了，真实需求是"保存搜索条件"，而不是"搜索功能"本身
   - 关键：每个问题附带一个猜测，猜错比猜对更有价值（用户会主动纠正）

3. **技术讨论**（1-2个问题，用 AskUserQuestion）
   - 覆盖：技术栈选择、代码风格偏好、影响范围评估
   - 优先提供选项而非开放式问题

4. **提出方案**（2-3个，推荐一个）
   - 必须包含一个"最小改动方案"
   - 明确标注"不做的事"（Not Doing）

5. **写需求文档**（保存到 `.buildrail/idea/YYYY-MM-DD-<topic>-requirement.md`）

6. **用户确认**
   - 确认后状态改为 APPROVED
   - 提示："需求文档已保存到 [路径]。是否开始后续流程？"

**异常处理**：
- 用户意图不明确 → 最多追问 3 次，之后给出最佳猜测让用户确认或纠正
- 影响范围超出预期 → 建议升级到大方向模式（`br-office-hours`）

**输出格式**（.buildrail/idea/YYYY-MM-DD-<topic>-requirement.md）：
```markdown
# 需求文档：<标题>

生成时间：YYYY-MM-DD
模式：小功能探索
状态：DRAFT / APPROVED

## 意图概述
（用户真正想要什么，2-3句话）

## 具体需求
## 技术方案
## 影响范围
## 不做的事情（Not Doing）
## 验收标准
```

## Recommended Approach

混合方案 C：复用核心提问逻辑，自建框架层。

**理由**：
- gstack office-hours 的追问模式（6个强制问题）经过大量用户验证，重写没意义
- superpowers brainstorm 的流程设计简洁高效，适合小功能场景
- agent-skills interview-me 的"guess + question"技巧是挖掘真实意图的好方法
- 但框架层（输出格式、交互风格、目录结构）按 BuildRail 风格重新设计，不受原项目约束

## 文档生命周期

- **DRAFT**：初始状态，文档还在讨论中
- **APPROVED**：用户明确确认（"同意"/"没问题"/"确认"），文档锁定
- **未完成**：用户中途放弃，保存当前进度并标注
- 状态变更不记录审批人/时间（个人工具，不需要审计）
- 回退规则：用户在确认前随时可以要求修改任何部分

## 下游对接契约

后续命令（如 `/spec` `/plan`）通过以下方式发现 idea 输出：
1. 扫描 `.buildrail/idea/` 目录下最新的 APPROVED 文件
2. 通过文件名后缀区分类型：`-design.md` = 设计文档，`-requirement.md` = 需求文档
3. 如果存在多个 APPROVED 文件，选择最新修改时间的那个

## shared/output-format.md 的职责

定义所有 BuildRail 输出文档的公共格式：
- YAML frontmatter 格式（生成时间、模式、状态）
- 状态枚举值（DRAFT / APPROVED / 未完成）
- 文件命名约定（`YYYY-MM-DD-<topic>-<type>.md`）
- 各 skill 的具体字段定义由各自的 SKILL.md 规定，shared/ 只定义公共部分

## Success Criteria

- `/idea` 能正确分流 70%+ 的场景（验证方式：准备 20 个标注好分类的测试输入，手动验证）
- `br-office-hours` 在 5-8 个追问内产出可用的设计文档
- `br-brainstorming` 在 2-3 个问题内产出可用的需求文档
- 两个 skill 都能独立使用，不依赖 `/idea`
- 所有交互和输出都是中文
