# 两条路径：全自动 vs 分步

> **为什么有这个文件**：BuildRail 服务两种用户——新手懒人（要"丢一句话全自动跑完"）和资深用户（要对每一步有掌控）。这两种诉求不该用一个"开关参数"硬凑在一起，而应该用**两个不同的入口**满足。斜杠命令（`/`）的心智模型是"命令名 + 说人话"，塞 `--auto`/`--manual` 参数既反直觉又记不住。所以 BuildRail 不用开关，用两条路径。

---

## 路径 A：全自动（br-full-dev）

```
/br-full-dev 帮我开发登录注册功能
```

- **零参数、零门禁、一路跑到底**
- 设计 → 计划 → 执行 → 审查 → 发布，一气呵成，不问不等
- scope-check 发现的 HIGH 问题**自动批量处理**（有推荐就改、没推荐就标记 tradeoff）
- 跑完用 `/br-status` 看它替你做了什么决策
- **唯一会停的情况**：连续 3 次验收失败（已超出"自动"能解决的范围，必须停，让用户介入）

**适合谁**：新手懒人、明确的小到中等功能、不想被打断。

**不适合谁**：支付/认证等敏感改动、想审查每行代码、资深用户。

---

## 路径 B：分步（底层命令）

```
/idea 帮我开发登录注册功能      → 做完输出"✅ 下一步可 /br-plan"
/br-plan                        → 做完输出"✅ 下一步可 /br-scope-check（可选）或 /run"
/br-scope-check                 → 做完输出"✅ 下一步可 /run"（可选）
/run                            → 做完输出"✅ 下一步可 /br-review"
/br-review                      → 做完输出"✅ 下一步可 /br-ship"
/br-ship                        → 发布完成
```

- 每个命令做完**自然停**——不是门禁打断，是这个命令的职责到这就结束了
- 做完**主动提示下一步**（解决"新手不知道接下来该调哪个"）
- 用户在任何一步都能介入、修改、重跑、跳过某步
- 资深用户可以不跑 scope-check、可以手动改计划再 run

**适合谁**：资深用户、敏感改动、想掌控每步、想学习 BuildRail 的流程。

**核心心智**：分步不是"被打断"，而是"这个命令做完了，接力棒交回给你"。

---

## 两条路径的对应关系

| | 路径 A（全自动） | 路径 B（分步） |
|---|---|---|
| 入口 | `/br-full-dev <需求>` | `/idea` → `/br-plan` → `/run` → ... |
| 门禁 | **无**，一路跑到底 | 无门禁，命令边界自然停 |
| 用户介入时机 | 跑完一次性看 `/br-status` | 每步之间 |
| scope-check HIGH | 自动批量处理 | 用户单独调时逐条审视 |
| 失败处理 | 自动重试，3 次后停 | 用户自己决定怎么修 |
| 参数 | 无 | 无 |
| 适合 | 新手懒人 | 资深 / 敏感改动 |

---

## 各 skill 如何判断自己在哪条路径

skill 本身**不需要读"模式"**。它通过**调用者**判断：

- **被 br-full-dev 级联调用** → 在路径 A 里 → 自动处理一切，不问用户
- **被用户直接调用**（`/br-plan`、`/br-scope-check` 等）→ 在路径 B 里 → 做完就停，提示下一步

判定方法：skill 正文里用"如果是由顶层编排器（如 `/br-full-dev`）调用的本流程"vs"如果被用户直接调用"来分支——这套约定在 BuildRail 里已存在（见 run.md、br-plan.md 的级联/独立调用区分），**不需要新增任何模式机制**。

> **scope-check 特殊处理**：它必须知道自己被谁调用，因为处理 HIGH 的策略不同。规则写在 `skills/br-scope-check/SKILL.md` 里：被 br-full-dev 调用 → 自动批量；被用户单独调 → 逐条 AskUserQuestion。

## 设计原则

1. **不发明开关**：用户的心智是"选哪个命令"，不是"给命令加参数"。
2. **门禁不属于全自动**：br-full-dev 的定义就是"无门禁"。需要门禁 = 需要掌控 = 用分步。
3. **分步的停顿是自然的**：不是打断，是职责边界。做完提示下一步，保持流畅。
4. **两条路径共享同一套 skill**：底层 skill（plan/run/verify/...）两边都用，区别只在"要不要等用户"。
5. **可观测性不分路径**：state.json + `/br-status` 两条路径都记录、都能查。
